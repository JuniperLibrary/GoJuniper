# 14 — 泛型

## 是什么

Go 1.18（2022 年）引入的泛型允许函数和类型通过**类型参数**（type parameter）对多种类型复用。语法上使用方括号 `[T any]` 而不是尖括号 `<T>`。约束（constraint）用接口表达，`any` 是 `interface{}` 的别名。

## 怎么用

### 类型参数和约束

```go
// Map 把 []A 映射为 []B，A B 可以是任意不同类型
func Map[A any, B any](xs []A, f func(A) B) []B {
    out := make([]B, 0, len(xs))
    for _, x := range xs {
        out = append(out, f(x))
    }
    return out
}
```

> ⚠️ **注意**：**`[T any]` 只是"任意类型"，但 `any` 约束下无法使用 `>`、`<`、`+` 等运算符**——因为 `any` 不保证类型支持这些操作。要对值做比较/运算，必须用更具体的约束（如 `cmp.Ordered`、或自定义的 `Number` 联合约束，见本仓库 `genericsx` 的 `GetLargest` / `Add`）。

### 常见的泛型高阶函数

```go
// Filter 保留满足条件的元素
func Filter[T any](xs []T, pred func(T) bool) []T {
    out := make([]T, 0, len(xs))
    for _, x := range xs {
        if pred(x) {
            out = append(out, x)
        }
    }
    return out
}

// Reduce 归并
func Reduce[T any, Acc any](xs []T, acc Acc, f func(Acc, T) Acc) Acc {
    for _, x := range xs {
        acc = f(acc, x)
    }
    return acc
}
```

### 带约束的泛型：`cmp.Ordered`

```go
func GetLargest[T cmp.Ordered](xs []T) (T, bool) {
    if len(xs) == 0 {
        var zero T
        return zero, false  // Go 没有 Option，用 bool 表示存在性
    }
    largest := xs[0]
    for _, v := range xs[1:] {
        if v > largest {
            largest = v
        }
    }
    return largest, true
}
```

`cmp.Ordered` 是 Go 1.21 新增的内置约束，覆盖所有支持 `< <= > >=` 的类型（int、float64、string、rune 等）。试用：

```go
GetLargest([]int{34, 50, 25, 100, 65})        // (100, true)
GetLargest([]rune{'y', 'm', 'a', 'q'})         // ('y', true)
GetLargest([]float64{3.14, 2.71, 1.618})       // (3.14, true)
```

> ⚠️ **注意**：Go 的运算符 `>` `<` **不是通过接口/trait 表达的**。`cmp.Ordered` 只是一个"标记接口"，编译器内置识别支持比较的类型；自定义类型**不能重载运算符**，所以 `GetLargest[MyStruct]` 用不了 `>`，除非为 `MyStruct` 自己写比较逻辑。这与 Java 的 `T extends Comparable<T>`（用户可实现 `compareTo`）不同——Java 通过接口方法表达可比性。

> ⚠️ **注意**：空切片没有"最大元素"可言，Go 用 `(zero, false)` 表达"无值"——**没有 Java 的 `Optional<T>`**。约定是"用第二个 bool 表示存在性"，调用方必须检查 `ok`，否则拿到的是类型零值（`var zero T`），容易误用。

## 为什么

对比 Java 的泛型：

| 方面 | Java | Go |
|------|------|----|
| 引入时间 | 5（2004） | 1.18（2022） |
| 语法 | `<T>` / `<T extends Comparable<T>>` | `[T constraint]` |
| 边界约束 vs 接口约束 | `T extends Comparable<T>` | `T cmp.Ordered` |
| 零值表达 | `null` | `var zero T` |
| 单位类型 | `void` | 无（返回 `(T, bool)` 模拟 Optional）|
| 类型参数方法 | 支持 `class Foo<T>` | 支持（1.18+） |
| 类型擦除 vs 单态化 | 类型擦除（运行时无类型信息） | 编译期单态化 |

核心差异在于 Java 的泛型是**类型擦除**（运行时无泛型类型信息，靠编译期检查），约束用 `extends`/`super` 边界；Go 的泛型是**编译期单态化**，约束系统等价于**接口**，表达能力比 Java 的边界约束弱：

- Go 没有关联类型（associated type），没有通配符 `? extends`/`? super`，没有复杂的 `where` 从句
- Go 的约束必须是接口（或 `any`），不能像 Java 那样用 `&` 多边界（Java 支持 `<T extends A & B>`，Go 需把多个方法塞进一个接口）
- Go 的运算符 `>` 不通过接口表达——只有编译器内置类型支持，自定义类型不能重载运算符。`cmp.Ordered` 只是一个标记接口，不是让用户实现 `compareTo` 接口的方式（Java 用 `Comparable.compareTo`）

## 常见坑

1. **泛型不能用于方法接收者的类型参数**（1.18 限制）：

```go
// 编译错误：方法不能有额外的类型参数
func (s *MySlice[T]) Map[U any](f func(T) U) []U { ... }
```

2. **`any` 不是无约束**：`[T any]` 等价于 `[T interface{}]`，可以传任意类型，但无法使用 `>`、`+` 等运算符，只能调用 `T` 接口上声明的方法。

3. **没有 `Optional<T>`**：Go 惯用 `(T, bool)` 或指针 + nil 表达可选值，不支持 Java 的 `Optional<T>` 模式匹配。

4. **性能认知**：Go 泛型是单态化的，运行时性能与手写特定类型版本一致，但编译产物会膨胀（代码膨胀问题类似 C++ template）。

5. **约束必须能被编译器验证**：写了 `[T cmp.Ordered]` 却传不支持比较的类型，编译期就报错；而只写 `[T any]` 却在函数体内用 `>`，也要到编译期才暴露。约束写错会直接编译失败，而非运行时。
