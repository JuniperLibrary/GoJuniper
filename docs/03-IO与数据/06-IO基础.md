# 06 — IO 基础：Reader / Writer / bufio

## 是什么

Go 把一切输入输出抽象成两个接口：`io.Reader`（读）和 `io.Writer`（写）。只要你实现了这两个接口，不管底层是文件、网络连接、内存 buffer 还是字符串，调用方代码都**不需要改**。

核心接口定义：

```go
type Reader interface {
    Read(p []byte) (n int, err error)
}
type Writer interface {
    Write(p []byte) (n int, err error)
}
```

`Read` 把数据填进你提供的切片 `p`，返回实际填充的字节数 `n`；`err == io.EOF` 表示读完了。`Write` 把 `p` 的数据写出去，返回实际写入的字节数。

## 怎么用

用 `io.ReadAll` 一口气读完：

```go
func ReadAllString(r io.Reader) (string, error) {
    b, err := io.ReadAll(r)
    if err != nil {
        return "", err
    }
    return string(b), nil
}
```

用 `bufio.Scanner` 按行读：

```go
func ReadLines(r io.Reader) ([]string, error) {
    sc := bufio.NewScanner(r)
    sc.Buffer(make([]byte, 0, 64*1024), 1024*1024)

    var lines []string
    for sc.Scan() {
        lines = append(lines, sc.Text())
    }
    if err := sc.Err(); err != nil {
        return nil, err
    }
    return lines, nil
}
```

用 `bufio.Writer` 缓冲写入，最后 `Flush()` 确保数据落地：

```go
func WriteLines(w io.Writer, lines []string) error {
    bw := bufio.NewWriter(w)
    for _, line := range lines {
        if _, err := fmt.Fprintln(bw, line); err != nil {
            return err
        }
    }
    return bw.Flush()
}
```

> ⚠️ **注意**：`bufio.Writer` **不会自动 Flush**（没有 RAII drop 机制）。缓冲的数据在 `Flush()` 之前可能根本没写出去。忘记调用 `Flush()` 是最常见的 IO bug 之一——务必在返回前调用（或用 `defer`，但注意顺序）。

> ⚠️ **注意**：`bufio.Scanner` **默认单行上限 64KB**。遇到超长行（如超大日志行）会直接报错 `token too long`。工程中遇到大行时必须手动 `sc.Buffer()` 扩容（如本仓库 `ReadLines` 所示），否则读取失败。

## 为什么（对比 Java）

| Go | Java | 差异要点 |
|----|------|----------|
| `io.Reader` | `java.io.InputStream` / `Readable` | 都只有一个读取方法。但 Java 的 `read()` 返回 `int`（字节数或 -1 表示 EOF），Go 通过 `(n int, err error)` 双返回值表达"读了 n 字节，但有错误（含 EOF）" |
| `io.ReadAll` | `InputStream.readAllBytes()` | 功能基本一致：读完整个流 |
| `bufio.Scanner` | `BufferedReader.readLine()` | Go 的 Scanner 默认有 token 长度限制（64KB），超长行需要手动 `Buffer()` 扩容；Java 的 `readLine()` 一般无此限制 |
| bufio 手动 Flush | `BufferedWriter` 需手动 `flush()` / `close()` | Go 的 bufio.Writer 在丢弃前**不会自动** Flush，忘记调用 Flush 是最常见的 bug 之一；Java 同样需要显式 `flush()`/`close()`（可用 try-with-resources） |

> ⚠️ **注意**：**`io.EOF` 不是"错误"**。它只是 `Read` 返回的"正常读完"信号。`io.ReadAll` 已经帮你处理了 EOF；但如果你自己写 `Read` 循环，EOF 表示正常结束，不要把它当异常处理或向上抛成错误。

Go 的核心哲学：**接口是约定，不是继承**。任何类型只要实现了 `Read(p []byte) (int, error)`，就自动是 `io.Reader`，不需要声明"implements"。Java 则需要显式 `implements` 接口，并实现其方法。

## 常见坑

1. **忘了 `Flush()`**：`bufio.Writer` 会在内存缓冲数据，不 Flush 的话数据根本没写出去。
2. **Scanner 默认 buffer 限制**：如果一行超过 64KB，Scanner 会报错。工程中遇到大行时，记得调 `sc.Buffer()`。
3. **忽略 `sc.Err()`**：`sc.Scan()` 返回 false 不代表全读完了，也可能是中间出错。`sc.Err()` 必须检查。
4. **`io.EOF` 不是"错误"**：`io.ReadAll` 帮你处理了 EOF，但如果你自己用 `Read` 循环，EOF 表示正常结束，不要让调用方把它当异常。
