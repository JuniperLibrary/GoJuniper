# 10 — 并发基础

## 是什么

goroutine 是 Go 的轻量级线程，由 Go runtime 调度而非 OS。启动一个 goroutine 只需在函数调用前加 `go` 关键字。goroutine 间的通信通过 channel 完成。

## 怎么用

### 启动 goroutine

```go
go func() {
    // 并发执行的代码
}()
```

> ⚠️ **注意**：goroutine **不会随启动它的函数返回而自动结束**。如果它在 channel 上阻塞且永远没人发送/接收，就会永久挂起造成**泄漏**。务必确保每条 goroutine 都有明确的退出路径（如监听 `ctx.Done()` 或 channel 关闭）。

### Worker Pool 模式

`internal/10-concurrency` 实现了可取消的 worker pool（`pool.go`）：

```go
type Job func(context.Context) error

func Run(ctx context.Context, concurrency int, jobs []Job) error
```

- 启动 N 个 worker goroutine，从 jobCh 读取任务
- 任一任务返回 error，通过 errCh 传递错误并调用 cancel() 通知其它 worker
- 生产者 goroutine 投递任务，遇到取消则停止

关键设计：

```go
// worker：从 jobCh 取任务执行，出错则发送到 errCh 并取消
go func() {
    defer wg.Done()
    for job := range jobCh {
        if err := job(ctx); err != nil {
            select {
            case errCh <- err:
                cancel()  // 通知其它 worker 停止
            default:      // errCh 已满（已有错误），不阻塞
            }
            return
        }
    }
}()

// 生产者：投递任务，支持取消
go func() {
    defer close(jobCh)
    for _, job := range jobs {
        select {
        case <-ctx.Done():
            return
        case jobCh <- job:
        }
    }
}()
```

> ⚠️ **注意**：**首错取消要用非阻塞写入 errCh**（`select { case errCh<-err: default: }`）。errCh 容量设为 1，第一个错误写入后其余错误走 `default` 分支直接丢弃、不阻塞——否则多个 worker 同时报错时会相互阻塞，导致 goroutine 卡死。

> ⚠️ **注意**：**WaitGroup 的 Add 必须在 Wait 之前、且在启动 goroutine 之前调用**（`wg.Add(concurrency)` 在 `go func(){...}()` 循环前）。规则经验：知道并发数就先 `Add`，goroutine 第一行 `defer wg.Done()`，最后 `wg.Wait()`。Add/Done 不匹配会死锁或 panic。

## 为什么

对比 Java：

| 特性 | Go | Java |
|------|----|------|
| 线程模型 | goroutine（M:N 调度） | OS 线程（1:1）或虚拟线程（Project Loom） |
| 栈大小 | 起步约 4KB，可动态增长 | 默认约 1MB（OS 线程）；虚拟线程栈极小 |
| 通信 | channel（CSP 模型） | `BlockingQueue`（或 `CompletableFuture`） |
| 取消 | `context.Context` | 无语言级取消机制，通常用 `Thread.interrupt()` / `Future.cancel()` |

Go 的 goroutine 是 **M:N 调度**：M 个 goroutine 映射到 N 个 OS 线程。相比 Java 的 1:1 平台线程模型，goroutine 启动成本极低（微秒级 vs 毫秒级）；Java 19+ 的虚拟线程（Project Loom）也实现了轻量 M:N 调度，适合大量并发任务（数十万级别）。

Java 用 `synchronized` / `ReentrantLock` + 共享对象引用（`volatile`/不可变对象）共享数据，Go 提倡"通过通信共享内存"——但也并非禁止互斥锁，实际中 `sync.Mutex` 同样广泛使用。

Go 的 `context.Context` 是标准化的取消/超时传播机制，Java 生态没有统一的语言级方案（常用 `Thread.interrupt()` 或 `Future.cancel()`）。

## 常见坑

1. **goroutine 泄漏**：goroutine 在 channel 上阻塞但没人发送/接收数据，导致永久挂起。务必确保 goroutine 能退出。

2. **循环变量捕获**：goroutine 闭包中直接使用循环变量会导致所有 goroutine 看到同一个值：
   ```go
   for _, v := range vals {
       go func() {
           fmt.Println(v) // 所有 goroutine 可能都打印最后一个值
       }()
   }
   ```
   正确做法：传参或拷贝：
   ```go
   for _, v := range vals {
       v := v  // 拷贝
       go func() { fmt.Println(v) }()
   }
   ```

3. **channel 未关闭导致死锁**：如果 `for range ch` 的 channel 不会被关闭，range 将永远阻塞。

4. **不检查 ctx.Err()**：长时间任务应定期检查 ctx.Done()，否则取消信号无法被响应。
