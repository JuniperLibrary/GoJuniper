# 10 — 并发基础

## 是什么

goroutine 是 Go 的轻量级线程，由 Go runtime 调度而非 OS。启动一个 goroutine 只需在函数调用前加 `go` 关键字。goroutine 间的通信通过 channel 完成。

## 怎么用

### 启动 goroutine

```go
go func() {
    // 并发执行的代码
}()
```

### Worker Pool 模式

`internal/concurrency` 实现了可取消的 worker pool（`pool.go`）：

```go
type Job func(context.Context) error

func Run(ctx context.Context, concurrency int, jobs []Job) error
```

- 启动 N 个 worker goroutine，从 jobCh 读取任务
- 任一任务返回 error，通过 errCh 传递错误并调用 cancel() 通知其它 worker
- 生产者 goroutine 投递任务，遇到取消则停止

关键设计：

```go
// worker：从 jobCh 取任务执行，出错则发送到 errCh 并取消
go func() {
    defer wg.Done()
    for job := range jobCh {
        if err := job(ctx); err != nil {
            select {
            case errCh <- err:
                cancel()  // 通知其它 worker 停止
            default:      // errCh 已满（已有错误），不阻塞
            }
            return
        }
    }
}()

// 生产者：投递任务，支持取消
go func() {
    defer close(jobCh)
    for _, job := range jobs {
        select {
        case <-ctx.Done():
            return
        case jobCh <- job:
        }
    }
}()
```

## 为什么

对比 Rust：

| 特性 | Go | Rust |
|------|----|------|
| 线程模型 | goroutine（M:N 调度） | OS 线程（1:1）或 async task |
| 栈大小 | 起步约 4KB，可动态增长 | 默认 2MB（OS 线程） |
| 通信 | channel（CSP 模型） | `std::sync::mpsc`、`crossbeam` |
| 取消 | `context.Context` | 无标准取消机制，通常用 `CancellationToken` |

Go 的 goroutine 是 **M:N 调度**：M 个 goroutine 映射到 N 个 OS 线程。相比 Rust 的 1:1 线程模型，goroutine 启动成本极低（微秒级 vs 毫秒级），适合大量并发任务（数十万级别）。

Rust 用 `Arc<Mutex<T>>` 共享数据，Go 提倡"通过通信共享内存"——但也并非禁止互斥锁，实际中 `sync.Mutex` 同样广泛使用。

Go 的 `context.Context` 是标准化的取消/超时传播机制，Rust 生态没有统一方案（常用第三方库如 `tokio_util::CancellationToken`）。

## 常见坑

1. **goroutine 泄漏**：goroutine 在 channel 上阻塞但没人发送/接收数据，导致永久挂起。务必确保 goroutine 能退出。

2. **循环变量捕获**：goroutine 闭包中直接使用循环变量会导致所有 goroutine 看到同一个值：
   ```go
   for _, v := range vals {
       go func() {
           fmt.Println(v) // 所有 goroutine 可能都打印最后一个值
       }()
   }
   ```
   正确做法：传参或拷贝：
   ```go
   for _, v := range vals {
       v := v  // 拷贝
       go func() { fmt.Println(v) }()
   }
   ```

3. **channel 未关闭导致死锁**：如果 `for range ch` 的 channel 不会被关闭，range 将永远阻塞。

4. **不检查 ctx.Err()**：长时间任务应定期检查 ctx.Done()，否则取消信号无法被响应。
