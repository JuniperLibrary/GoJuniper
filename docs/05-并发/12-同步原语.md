# 12 — 同步原语

## 是什么

Go 标准库 `sync` 包提供基于共享内存的同步原语：`Mutex`（互斥锁）、`RWMutex`（读写锁）、`Once`（只执行一次）、`WaitGroup`（等待一组 goroutine）、`Cond`（条件变量）。

## 怎么用

`internal/12-syncx` 演示了 `Mutex` 和 `Once` 两种典型用法（`syncx.go`）：

### sync.Mutex — 保护共享变量

```go
type Counter struct {
    mu sync.Mutex
    n  int
}

func (c *Counter) Inc() {
    c.mu.Lock()
    c.n++
    c.mu.Unlock()
}

func (c *Counter) Value() int {
    c.mu.Lock()
    v := c.n
    c.mu.Unlock()
    return v
}
```

> ⚠️ **注意**：Go 的 `sync.Mutex` **没有 RAII 守卫，必须手动 `Unlock`**。忘记解锁会死锁。标准写法是用 `defer mu.Unlock()` 紧跟 `Lock()`，确保任何 return 路径（包括 panic）都能释放锁——这也是本仓库 `Counter` 之外我建议统一采用的写法（源码里的 `Inc` 直接 `Unlock` 仅作演示，生产代码请加 `defer`）。

> ⚠️ **注意**：**`sync.Mutex` 不能被复制/拷贝**。含 `Mutex` 的结构体必须用**指针接收者**（`func (c *Counter)`），否则复制出的副本是独立的锁状态，保护彻底失效。`go vet` 能检测这种"值拷贝 Mutex"错误。

### sync.Once — 只执行一次

```go
type OnceValue[T any] struct {
    once sync.Once
    v    T
}

func (o *OnceValue[T]) Get(f func() T) T {
    o.once.Do(func() {
        o.v = f()       // 仅第一次调用时执行
    })
    return o.v
}
```

测试中验证 100 个 goroutine 并发 Inc，最终结果正确：

```go
var c syncx.Counter
var wg sync.WaitGroup
wg.Add(100)
for i := 0; i < 100; i++ {
    go func() {
        defer wg.Done()
        c.Inc()
    }()
}
wg.Wait()
// c.Value() == 100
```

> ⚠️ **注意**：`sync.Once.Do(f)` 保证 `f` **并发安全地只执行一次**（懒加载的惯用手段）。`Once` 内部有状态，同样不能被拷贝，且 `Do` 只接受无参函数，需要传参时把参数闭包捕获进 `func(){}` 里。

## 为什么

对比 Java：

| 特性 | Go | Java |
|------|----|------|
| 互斥锁 | `sync.Mutex` | `ReentrantLock` / `synchronized` |
| 守卫对象 | 无，手动 Lock/Unlock（常用 defer） | `synchronized` 块（作用域结束自动释放）/ `ReentrantLock` + try-finally |
| 读写锁 | `sync.RWMutex` | `ReentrantReadWriteLock` |
| 一次性初始化 | `sync.Once` | 静态初始化 / `holder` 惯用法（lazy init holder） |
| 条件变量 | `sync.Cond` | `Object.wait()` / `Condition` |
| 编译期检查 | 无 | 无（数据竞争靠 `volatile` / 锁约定，运行时可用 `-Xlint`/工具检测） |

**核心差异**：Java 的 `synchronized` 通过语言结构保证锁的获取/释放（块级作用域），`ReentrantLock` 需配合 try-finally 释放。Go 的 `sync.Mutex` 只是一个锁，保护哪段代码完全靠约定（通常用 `defer` 释放）——编译器和运行时都不会帮你检查。

Go 的 `sync.Once` 使用比 Java 的静态 holder 惯用法更灵活：不是包装一个值，而是包装一个闭包执行逻辑。它结合了 "只执行一次" 的语义 + 任意副作用。

Go 没有 RAII，忘记 `Unlock` 会导致死锁。实践中常用 `defer mu.Unlock()` 确保解锁。

**为什么 Go 没有 `Arc` 那样的引用计数**？Go 有 `sync.Mutex` 但不需要 `Arc`——Go 的 goroutine 共享同一地址空间（堆上的变量可以被任意 goroutine 访问），引用计数由 GC 管理，无需手动 `clone`。

## 常见坑

1. **忘记解锁**：`Lock()` 后忘记 `Unlock()`，或某条路径提前 return 跳过了解锁。务必用 `defer`：
   ```go
   mu.Lock()
   defer mu.Unlock()
   ```

2. **拷贝 sync.Mutex**：`sync.Mutex` 不能拷贝（拷贝后两者不共享锁状态）。结构体包含 `sync.Mutex` 时必须用指针接收者。`go vet` 能检测这个问题。

3. **Mutex 与 channel 的选择**：保护共享变量用 Mutex；goroutine 间传递数据用 channel。不要在一个问题上混合使用两种方式。

4. **RWMutex 误用**：读多写少用 `RWMutex` 提升并发读性能。但写锁会阻塞所有读锁，读锁也会阻塞写锁。写锁优先级可能低于读锁，导致写锁饥饿。

5. **WaitGroup Add/Done 不匹配**：`Add` 必须在 `Wait` 之前调用，`Done` 必须在 goroutine 结束前调用。推荐在知道 goroutine 数就 Add，在 goroutine 第一行用 `defer wg.Done()`。
