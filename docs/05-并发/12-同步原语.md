# 12 — 同步原语

## 是什么

Go 标准库 `sync` 包提供基于共享内存的同步原语：`Mutex`（互斥锁）、`RWMutex`（读写锁）、`Once`（只执行一次）、`WaitGroup`（等待一组 goroutine）、`Cond`（条件变量）。

## 怎么用

`internal/12-syncx` 演示了 `Mutex` 和 `Once` 两种典型用法（`syncx.go`）：

### sync.Mutex — 保护共享变量

```go
type Counter struct {
    mu sync.Mutex
    n  int
}

func (c *Counter) Inc() {
    c.mu.Lock()
    c.n++
    c.mu.Unlock()
}

func (c *Counter) Value() int {
    c.mu.Lock()
    v := c.n
    c.mu.Unlock()
    return v
}
```

### sync.Once — 只执行一次

```go
type OnceValue[T any] struct {
    once sync.Once
    v    T
}

func (o *OnceValue[T]) Get(f func() T) T {
    o.once.Do(func() {
        o.v = f()       // 仅第一次调用时执行
    })
    return o.v
}
```

测试中验证 100 个 goroutine 并发 Inc，最终结果正确：

```go
var c syncx.Counter
var wg sync.WaitGroup
wg.Add(100)
for i := 0; i < 100; i++ {
    go func() {
        defer wg.Done()
        c.Inc()
    }()
}
wg.Wait()
// c.Value() == 100
```

## 为什么

对比 Rust：

| 特性 | Go | Rust |
|------|----|------|
| 互斥锁 | `sync.Mutex` | `std::sync::Mutex<T>` |
| 守卫对象 | 无，手动 Lock/Unlock | `MutexGuard`（RAII，作用域结束自动释放） |
| 读写锁 | `sync.RWMutex` | `std::sync::RwLock<T>` |
| 一次性初始化 | `sync.Once` | `std::sync::OnceLock`、`lazy_static` |
| 条件变量 | `sync.Cond` | `std::sync::Condvar` |
| 编译期检查 | 无 | 借用检查器（数据竞争在编译期被拒绝） |

**核心差异**：Rust 的 `Mutex<T>` 通过类型系统保证内部值只能通过锁访问（Poisoning 机制也能检测 panic）。Go 的 `sync.Mutex` 只是一个锁，保护哪段代码完全靠约定——编译器和运行时都不会帮你检查。

Go 的 `sync.Once` 使用比 Rust 的 `OnceLock` 更灵活：不是包装一个值，而是包装一个闭包执行逻辑。它结合了 "只执行一次" 的语义 + 任意副作用。

Go 没有 RAII，忘记 `Unlock` 会导致死锁。实践中常用 `defer mu.Unlock()` 确保解锁。

**为什么 Go 没有 `Arc` 那样的引用计数**？Go 有 `sync.Mutex` 但不需要 `Arc`——Go 的 goroutine 共享同一地址空间（堆上的变量可以被任意 goroutine 访问），引用计数由 GC 管理，无需手动 `clone`。

## 常见坑

1. **忘记解锁**：`Lock()` 后忘记 `Unlock()`，或某条路径提前 return 跳过了解锁。务必用 `defer`：
   ```go
   mu.Lock()
   defer mu.Unlock()
   ```

2. **拷贝 sync.Mutex**：`sync.Mutex` 不能拷贝（拷贝后两者不共享锁状态）。结构体包含 `sync.Mutex` 时必须用指针接收者。`go vet` 能检测这个问题。

3. **Mutex 与 channel 的选择**：保护共享变量用 Mutex；goroutine 间传递数据用 channel。不要在一个问题上混合使用两种方式。

4. **RWMutex 误用**：读多写少用 `RWMutex` 提升并发读性能。但写锁会阻塞所有读锁，读锁也会阻塞写锁。写锁优先级可能低于读锁，导致写锁饥饿。

5. **WaitGroup Add/Done 不匹配**：`Add` 必须在 `Wait` 之前调用，`Done` 必须在 goroutine 结束前调用。推荐在知道 goroutine 数就 Add，在 goroutine 第一行用 `defer wg.Done()`。
