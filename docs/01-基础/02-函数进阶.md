# 02 — 函数进阶

## 是什么

Go 函数是一等公民：可作为参数、返回值，可赋值给变量。闭包、defer、变参、panic/recover 是 Go 函数进阶的核心特性。

## 怎么用

### 闭包（closure）

闭包捕获外层变量，类似 Java 的 lambda 表达式（Java lambda 捕获等效 final 的局部变量）：

```go
func Counter() func() int {
    var count int          // 被闭包捕获
    return func() int {
        count++            // 修改捕获的变量
        return count
    }
}

c := Counter()
c() // 1
c() // 2
// 每个 Counter() 调用创建独立的闭包状态
```

参见 `internal/02-funcsx/funcsx.go:12` 的 `Counter` 实现。

> ⚠️ **注意**：Go 闭包**默认按引用捕获**外层变量。当多个 goroutine / 多次调用共享同一个闭包时，它们操作的是同一个变量。如果需要"每次循环独立一份"，要显式拷贝（如 `for _, v := range xs { v := v; go ... }`），否则所有闭包会看到同一个最终值。

### defer — 延迟执行

defer 在函数返回前执行，后进先出（LIFO）：

```go
func DeferInspect(nums []int) (log string) {
    for _, n := range nums {
        defer func() {
            log += fmt.Sprintf("defer(%d)", n)
        }()
    }
    log += "body"
    return
}
// 结果："bodydefer(3)defer(2)defer(1)"
```

参见 `internal/02-funcsx/funcsx.go:44`。

> ⚠️ **注意**：**defer 的参数在注册时求值，但闭包里引用的变量延迟到执行时才求值**。例如 `defer fmt.Print(x)` 在注册时就固定了 x 的值；而 `defer func(){ fmt.Print(x) }()` 捕获的是变量 x，执行时才读。理解这个差异才能预测 defer 输出的内容。

### 变参函数（variadic）

```go
func Sum(nums ...int) int {
    total := 0
    for _, n := range nums {
        total += n
    }
    return total
}

Sum(1, 2, 3)          // 多个参数
Sum(nums...)          // 展开切片
Sum()                 // 空调用 → 0
```

参见 `internal/02-funcsx/funcsx.go:33`。

> ⚠️ **注意**：变参 `...int` **必须放在参数列表最后一位**，且一个函数最多只能有一个变参。展开切片调用时必须加 `...`：`Sum(nums...)`，不能直接传 `Sum(nums)`。

### panic / recover

panic 类似 Java 未捕获异常（JVM 抛 `RuntimeException`），recover 类似 `try-catch(Throwable)` 兜底：

```go
func RecoverFromPanic(f func()) (err error) {
    defer func() {
        if r := recover(); r != nil {
            err = fmt.Errorf("panicked: %v", r)
        }
    }()
    f()
    return nil
}
```

参见 `internal/02-funcsx/funcsx.go:22`。

> ⚠️ **注意**：`recover()` **只能在 `defer` 函数里调用才有效**；放在别处返回 `nil`。同时它能**改写命名返回值**——上面 `RecoverFromPanic` 用 `(err error)` 命名返回，defer 里赋值后能改变最终返回结果，这是 Go 的 recover 标准套路。

## 为什么（对比 Java）

- **Go 闭包默认可变捕获**：Java lambda 只能捕获 `final` / 等效 final 的局部变量，且不能修改；Go 闭包默认捕获变量的引用，可直接读写，无需额外标记。Go 没有生命周期标注，闭包用起来"更省事"，但无法在编译期防止悬垂引用。
- **defer ≈ Java try-with-resources（AutoCloseable）**：defer 是函数返回前的显式清理，Java 用 `try (resource) {}` 在块结束时自动调用 `close()`。Go 的 defer 需要手写，但更灵活（比如能修改返回值）。
- **defer 在函数作用域结束才执行**，不像 Java 的 try-with-resources 在块级（花括号）结束时执行。Go 没有块级 defer。
- **没有 `?` 运算符 / 受检异常**：Go 用 `if err != nil` 手动传播错误。Java 用 `try-catch` 或（非受检）异常传播，函数调用链长时各有繁简。
- **panic/recover 仅用于"不可能发生的"错误**，类似 Java 中 `throw new RuntimeException()` 的兜底。正常错误用 `error` 返回值（Java 用 `Exception`）。

## 常见坑

1. **defer 参数立即求值，闭包变量延迟求值**：
   ```go
   // 错误：defer 时已求值 x=0，最终全是 0
   for x := 0; x < 3; x++ {
       defer fmt.Print(x)
   }
   // 正确：闭包捕获 x，发现 2,1,0（LIFO）
   for x := 0; x < 3; x++ {
       defer func() { fmt.Print(x) }()
   }
   ```
   Go 1.22 后 `for` 循环变量重建，上述问题已缓解。但仍需理解 defer 求值时机。

2. **recover 只能在 defer 中有效** — 在 `defer` 函数外调用 `recover()` 返回 `nil`。

3. **panic 会越过所有 defer 执行 recover** — 如果 defer 中的 recover 没有命中，panic 继续向上传播。

4. **`for` 中 defer 累积不会在每次迭代后执行** — 所有 defer 在包含它们的函数返回前才执行。循环中大量使用 defer 会导致资源延迟释放（defer 栈膨胀）。

5. **变参展开切片时不能传 `[]int` 本身** — 需要加 `...`：`Sum(nums...)`。
