# 04 — 类型系统

## 是什么

Go 的类型系统以 struct（结构体）、method（方法）、interface（接口）和
embedding（组合）为核心。没有类（class）和继承（inheritance），
用**组合取代继承**。

## 怎么用

### struct 与方法

```go
type User struct {
    ID   int
    Name string
}

// 值接收者：不会修改原对象
func (u User) Greeting() string {
    return fmt.Sprintf("hello, %s", u.Name)
}

// 指针接收者：会修改原对象
func (u *User) SetName(name string) error {
    if u == nil {
        return errors.New("nil receiver")
    }
    u.Name = name
    return nil
}
```

值接收者操作的是副本，指针接收者可以修改原对象。
两种接收者都可以调用，Go 会自动转换。

> ⚠️ **注意**：**值接收者 vs 指针接收者的选择影响接口实现**。如果接口方法用指针接收者实现，那么**值类型变量无法满足该接口变量**（需要取地址）。反过来值接收者实现的方法，值与指针都能满足接口。规则经验：需要修改接收者、或结构体较大避免拷贝时，统一用指针接收者。

### 构造函数

Go 没有构造函数的语法关键字，用普通函数代替。

```go
func NewUser(id int, name string) (User, error) {
    if id <= 0 {
        return User{}, ErrInvalidID
    }
    name = strings.TrimSpace(name)
    if name == "" {
        return User{}, ErrEmptyName
    }
    return User{ID: id, Name: name}, nil
}
```

### 接口

Go 的接口是隐式实现（duck typing）：
一个类型只要实现了接口的所有方法，就自动满足该接口，无需 `impl` 关键字。

```go
// 标准库中的 Stringer 接口
type Stringer interface {
    String() string
}

// User 自动实现了 Stringer
func (u User) String() string {
    return fmt.Sprintf("User{ID:%d, Name:%q}", u.ID, u.Name)
}
```

> ⚠️ **注意**：接口是**隐式实现**——没有 `impl` 声明。但有一个经典坑：**`var s fmt.Stringer = User{}` 没问题，但 `var s fmt.Stringer = nil` 接口本身不是 nil 语义**。更隐蔽的是"nil 接口调用方法 panic"：只要接口底层 `(type, value)` 的 value 是 nil，**对接口调用方法会 panic**；只有接口整体为 nil 时才安全。用指针接收者实现方法时尤需小心。

### 组合（Embedding）

Go 用 struct embedding 实现类似继承的效果，
但本质是组合：外层类型"提升"内层的字段和方法。

```go
type Admin struct {
    User          // 嵌入 User，自动获得其字段和方法
    Level int
}

a := Admin{User: User{ID: 1, Name: "alice"}, Level: 10}
a.Name        // 直接访问 User 的字段（被提升）
a.Greeting()  // 直接调用 User 的方法（被提升）
a.IsSuper()   // Admin 自己的方法
```

> ⚠️ **注意**：**embedding 不是继承，不能重写（override）方法**。嵌入后内层字段/方法只是被"提升"到外层，调用时仍然分发给内层实现。Go 没有多态分发或父类/子类概念，它只是语法糖层面的委托（delegation）。

## 为什么

对比 Java：

| 特性 | Go | Java |
|------|----|------|
| 自定义类型 | struct + method | class + method |
| 接口 | 隐式实现（duck typing） | 显式 `implements` |
| 组合 | embedding（字段/方法提升） | 手动委托（无匿名嵌入语法） |
| 继承 | 无 | 单继承 `extends`（类）/ 多 `implements`（接口） |
| 泛型 | Go 1.18+ 引入 | 从 5 就有，更完善 |

Java 的接口需要显式 `implements`，Go 的 interface 是隐式的——这让
Go 可以方便地为已有类型补充接口实现（比如为 `User` 加 `String()` 方法使其
自动满足 `fmt.Stringer`）。Java 则需修改类声明加 `implements`。

Go 的 embedding 不是继承。嵌入后无法重写（override）方法。
它是语法糖层面的委托（delegation）。Java 用显式方法转发实现委托。

## 常见坑

1. **指针接收者直接调用导致的 nil panic**：Go 允许用 nil 指针调用方法，
    所以在方法开头要检查 `if u == nil { ... }`。Java 的 NPE 也类似——对 null 引用调用方法会在运行时抛异常，编译期不阻止（除非用 `@Nullable` 注解 + 工具检查）。

2. **值接收者与接口**：如果接口方法用指针接收者实现，值类型变量无法通过
   接口变量调用该方法（需要取地址）。

3. **Embedding 不是继承**：嵌入类型的字段会被"提升"，但不能被"重写"，
   也没有多态分发的效果。

4. **nil 接口 vs nil 值的接口**：接口变量 `(T, nil)` 不等于 `nil` 接口，对前者调用方法会 panic。判断接口是否为空要用 `== nil`，而非依赖内部值。
