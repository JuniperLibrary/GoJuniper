## 1. struct（结构体）是什么

结构体就是把多个字段组合成一个新类型：

```go
// 来自 typesx.go
type User struct {
	ID   int
	Name string
}
```

创建方式：

```go
u1 := User{ID: 1, Name: "Alice"} // 推荐：字段名写清楚
u2 := User{}                     // 全零值
_, _ = u1, u2
```

> ⚠️ **注意**：结构体是**值类型**。把 `User` 赋值给另一个变量或当作参数传进去，会发生一次**整体拷贝**——修改副本不会影响原值。结构体很大时（很多字段）用值传递代价高，这时应该传 `*User`（指针）。

对应 Rust：`type User struct { ... }` ≈ `struct User { ... }`，都是命名结构体；但 Go 没有 Rust 那种 `impl` 块与 `derive`，方法要单独写在类型上。

## 2. 方法（method）是什么

方法就是“绑定在某个类型上的函数”：

```go
func (u User) Greeting() string {
	return fmt.Sprintf("hello, %s", u.Name)
}
```

### 2.1 值接收者 vs 指针接收者

你只需要先掌握两个规则：

- **想修改接收者本身**：用指针接收者 `func (u *User) SetName(...)`
- **类型比较大/避免拷贝**：也用指针接收者

如果方法只是读数据，值接收者也可以。

注意：当你写了指针接收者的方法，Go 也允许你用值来调用（编译器会自动取地址），反之亦然有一些限制，但初学阶段记住“修改就用指针”基本够用。

> ⚠️ **注意**：**值接收者改不了字段**。看 `typesx.go`：`Greeting()` 用值接收者 `(u User)`，只读 `u.Name`；`SetName()` 用指针接收者 `(u *User)`，会做 `u.Name = name` 真正改掉原对象。如果用值接收者去写 `u.Name = name`，改的只是副本，原对象纹丝不动——这是初学者最常见的“修改不生效”坑。

**为什么这么设计**：Go 通过接收者类型决定方法能否修改对象。值接收者保证“调用方法不会偷偷改我的数据”，指针接收者才拥有修改权；同时指针接收者也避免了大结构体的拷贝开销。对应 Rust：`&self`（只读）≈ 值接收者，`&mut self`（可改）≈ 指针接收者。

### 2.2 方法集（method set）：值 vs 指针满足接口的差异

> ⚠️ **注意（最容易被忽略的规则）**：一个类型**能否实现接口**，取决于它的**方法集**。
> - `User`（值）的方法集：**只包含值接收者**的方法
> - `*User`（指针）的方法集：**包含值接收者 + 指针接收者**的所有方法
>
> 所以如果你的接口方法是指针接收者（如 `Describe()`），那么 `*User` 满足该接口，但 `User` 值**不满足**。

```go
// 来自 typesx.go
type Describer interface {
	Describe() string
}

func (u *User) Describe() string { // 指针接收者
	return fmt.Sprintf("User{ID:%d, Name:%q}", u.ID, u.Name)
}

// AssignDescriber 演示：只有 *User 能赋给 Describer
func AssignDescriber(u User) bool {
	var d Describer = &u // OK：*User 满足 Describer
	_ = d
	return true
}
```

对应测试见：[typesx_typesx_test.go](../16-tests/typesx_typesx_test.go)（`TestAssignDescriber` 用 `reflect` 验证 `*User` 实现、`User` 值不实现 `Describer`）。

> ⚠️ **注意**：反过来也成立——如果你写了值接收者方法，值和指针**都**满足接口（因为指针可以自动解引用）。记住口诀：**指针接收者方法，只有指针满足接口；值接收者方法，值和指针都满足。** 这在把具体类型塞进接口变量、或作为函数参数传递时直接决定编译成败。

## 3. interface（接口）是什么

接口是一组方法的集合：

```go
type Stringer interface {
	String() string
}
```

### 3.1 Go 的接口是“隐式实现”

Go 里不需要写 `implements`。

只要一个类型拥有接口要求的方法，它就自动实现该接口。

这也是 Go 很推崇的小接口风格的原因：接口越小，越容易复用。

> ⚠️ **注意**：**接口零值是 `nil`**。对一个 `nil` 接口调用方法会直接 `panic`（`invalid memory address or nil pointer dereference`）。更隐蔽的坑是：**“持有 nil 指针的非 nil 接口”**——接口变量本身非 nil，但它内部装的指针是 nil，此时调用方法会进到接收者里，如果方法里没做 `u == nil` 判断就会 panic。`typesx.go` 的 `SetName` 特意写了 `if u == nil { return errors.New("nil receiver") }` 就是防这个。

> ⚠️ **注意**：**隐式实现 ≠ 继承/自动 derive**。Go 是“结构化类型（structural typing）”：只要方法签名对得上就满足接口，无需声明。Rust 的 trait 实现需要显式 `impl Trait for Type`，但也是结构化检查；区别在于 Go 连 `impl` 这一步都省了，编译期自动判定。

### 3.2 “含 nil 指针的非 nil 接口”陷阱

> ⚠️ **注意（最隐蔽的接口坑，教材《Learning Go》重点强调）**：接口变量由两部分组成——**类型信息 + 值（指针/数据）**。一个接口只有在“类型信息和值都为 nil”时才等于 `nil`。如果接口里装了一个 **nil 指针**，接口本身**不是 nil**，但底层指针是 nil。

```go
// 来自 typesx.go
type NilError struct{}

func (e *NilError) Error() string { return "nil error" }

func ReturnsNilError() error {
	var p *NilError = nil
	return p // 返回的是 *NilError(nil)，装入 error 接口后接口非 nil！
}

func ReturnsRealNil() error {
	return nil // 真正的 nil
}
```

对应测试见：[typesx_typesx_test.go](../16-tests/typesx_typesx_test.go)（`TestReturnsNilError`）。

> ⚠️ **注意**：`ReturnsNilError()` 返回的 `error` 接口，`err == nil` 判断是 **false**（接口非 nil），但底层 `*NilError` 是 nil。这种代码最危险——`if err != nil` 会误判"有错误"，实际调用 `err.Error()` 反而可能 panic（nil 指针解引用）。**正确做法**：永远返回真正的 `nil`（`return nil`），不要返回"装了 nil 指针的接口"。这是 Go 错误处理里排名第一的隐蔽 bug。

## 4. embedding（组合/嵌入）是什么

嵌入是一种“用组合代替继承”的做法：

```go
// 来自 typesx.go
type Admin struct {
	User  // 嵌入 User，不是普通字段
	Level int
}
```

你可以直接通过 `u.ID` / `u.Name` 访问嵌入字段（像“提升”了一样），但本质仍是组合。

> ⚠️ **注意**：**Go 没有继承，`Admin` 不是 `User`**。不能直接写 `var u User = admin`（编译错误）。嵌入只是把被嵌入类型的字段和方法“提升”到外层，方便调用；如果要当 `User` 用，要写 `admin.User`。这与 Rust 的“组合一个字段”概念一致，但 Go 用匿名嵌入省去了手写转发方法。

## 5. fmt.Stringer：一个最常见的接口例子

标准库里经常用 `fmt.Stringer`：

```go
type Stringer interface {
	String() string
}
```

`typesx.go` 里 `User` 实现了它：

```go
func (u User) String() string {
	return fmt.Sprintf("User{ID:%d, Name:%q}", u.ID, u.Name)
}
```

实现它后：
- `fmt.Println(x)` 会更漂亮
- 调试输出更清晰

对应实现请看：[typesx.go](../04-typesx/typesx.go)

> ⚠️ **注意**：只要有 `String() string` 方法，`fmt` 包在打印时就会自动调用它（隐式满足 `fmt.Stringer`）。这正是“小接口 + 隐式实现”组合的威力：你不用改 `fmt` 源码，只要自己的类型带上对应方法就生效。

## 6. 初学者常见坑

1. 用值接收者写“修改方法”，导致修改不生效
2. 以为接口需要显式声明实现（其实只要方法集匹配即可）
3. 把 embedding 当继承使用（Go 更推荐明确组合与委托，且 `Admin` 不能赋值给 `User`）
4. 对 `nil` 接口或“含 nil 指针的接口”直接调用方法导致 panic
