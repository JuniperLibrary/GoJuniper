## 0. 先建立一个“脑模型”

你可以把 Go 里的“名字”分成两大类：

- **变量（var / :=）**：运行期存在、可变、通常“有内存位置”
- **常量（const）**：编译期确定、不可变、很多时候只是“编译器替换”，不一定有内存位置

作用域就是：这些名字在什么范围内可见。

初始化就是：这些名字的值是按什么顺序被计算出来的。

## 1. 变量（var）是什么

变量（variable）就是“有名字的可变值”。Go 用 `var` 声明变量：

```go
var age int = 18
var name string = "Alice"
```

### 1.1 类型可写可不写（类型推断）

```go
var age = 18       // 推断为 int
var name = "Alice" // 推断为 string
```

### 1.2 零值（zero value）

Go 的变量只要声明了，一定有值（默认零值），不会像某些语言那样是“未初始化”。

常见零值：

- `int`/`float64`：`0` / `0.0`
- `bool`：`false`
- `string`：`""`
- 指针/切片/映射/channel/函数/interface：`nil`
- struct：每个字段都是零值（整体也有一个“全零”的状态）

例子：

```go
var a int             // 0
var b bool            // false
var s string          // ""
var p *int            // nil
var xs []int          // nil
var m map[string]int  // nil
```

零值很重要的原因：

- 你可以写出更少的初始化代码
- 很多类型零值就能直接用（例如 `var mu sync.Mutex`）

> ⚠️ **注意**：零值虽好，但 **nil map / nil chan / nil func / nil pointer 不能直接"用"**（写入/调用会 panic）。只有 slice 的零值可以安全 `append`（append 内部帮你初始化）。所以"声明即零值"不等于"声明即可用"，要看类型。

> ⚠️ **注意（Java 对照）**：Java 里局部变量 `int x;` 不初始化就编译报错，强制你给值（类字段才有默认零值）。Go 则一律给零值（局部变量也是）——方便，但也更容易写出"忘了初始化就直接用"的 bug，尤其是 nil map 写入 panic。

## 2. 变量声明的几种写法（你必须会）

### 2.1 单个 var

```go
var x int
var y int = 10
var z = 20 // 推断类型
```

### 2.2 var 组（常见于包级变量与结构化声明）

```go
var (
	host = "127.0.0.1"
	port = 8080
)
```

### 2.3 多重赋值（交换变量/同时返回）

```go
a, b := 1, 2
a, b = b, a
```

> ⚠️ **注意**：`a, b = b, a` 是**平行赋值**——等号右边先全部求值，再赋给左边。所以一行就能交换，不需要临时变量。这是 Go 的小语法糖，也是 slice/多返回值处理里最常用的惯用法。

## 3. 短变量声明（:=）

在函数内部，你最常用的是 `:=`：

```go
func demo() {
	x := 10
	name := "Alice"
	_, _ = x, name
}
```

### 3.1 `:=` 的限制

- 只能在**函数内部**使用（不能在包级/全局使用）
- 左侧至少有一个变量是“新声明”的（否则会报错）

```go
func demo() {
	x := 10
	x := 20 // 编译错误：no new variables on left side of :=
	_ = x
}
```

但这种是允许的（因为 `y` 是新变量，`x` 是重新赋值）：

```go
func demo() {
	x := 10
	x, y := 20, 30 // OK
	_, _ = x, y
}
```

> ⚠️ **注意（最容易误解的点）**：`x, y := 20, 30` 里 `x` 明明已经存在，为什么还合法？因为 `:=` 只要求**左侧至少有一个新变量**（`y` 是新的）。此时 `x` 是**被重新赋值**（同一个变量，不是新建），`y` 是新声明。这跟 Java 的变量重定义（同作用域不允许重复声明同名变量）**行为不同**：Go 这里是真改了原变量的值，不是新建绑定。见 `basics_test.go` 的 `TestReassignWithShort` 断言 `x==20, y==30`。

### 3.2 := 最容易踩的坑：遮蔽（shadowing）

你以为是在“给外层变量赋值”，其实声明了一个新变量，导致外层变量没变。

```go
func demo() error {
	var err error
	if true {
		err := doSomething() // 新变量 err，遮蔽了外层 err
		_ = err
	}
	return err // 返回外层 err，可能还是 nil
}
```

修复方法：把 `:=` 改成 `=`（前提是外层已声明）。

> ⚠️ **注意**：遮蔽（shadowing）是 Go 里极隐蔽的 bug 来源，尤其是 `err` 变量。在 `if`/`for` 块里用 `:=` 声明的 `err` 和外层 `err` 不是同一个，外层拿不到块内的值，还可能返回 nil 错误。Go 官方 `go vet` 能查出部分遮蔽，但工具不是万能——写 `:=` 前先想清楚"我要不要新变量"。

## 4. 作用域（scope）到底怎么理解

作用域就是“这个名字在代码的哪一段可见”。

Go 常见的作用域层级：

- **包级作用域**：在 `package xxx` 下、函数外声明的
- **函数级作用域**：函数体内声明的
- **块级作用域**：`{ ... }` 内声明的（例如 `if`/`for`/`switch`）

### 4.1 块作用域示例

```go
func demo() {
	x := 1
	{
		x := 2 // 新变量，遮蔽外层 x
		_ = x
	}
	_ = x // 这里仍然是 1
}
```

### 4.2 if/for 里常见的“局部声明”

```go
if v := compute(); v > 0 {
	// v 只在 if 的这个块里可见
}
```

```go
for i := 0; i < 10; i++ {
	// i 只在 for 循环体内可见
}
```

> ⚠️ **注意**：`if v := compute(); v > 0 { ... }` 里声明的 `v` **只在 if 块内可见**，包括 `else` 分支，但不包含 if 之后的代码。如果想在 if 外也用，先把变量声明在外面。

## 5. 全局变量（包级变量）

“全局变量”在 Go 里通常指**包级变量**（package-level variable），写在函数外面：

```go
package p

var GlobalCount int
```

### 5.1 包级变量的特点

- 生命周期：程序启动到退出一直存在
- 可见性（是否能被其它包使用）：看首字母大小写
  - `GlobalCount`（大写）：导出，可被其它包导入使用
  - `globalCount`（小写）：不导出，仅包内使用

> ⚠️ **注意（Java 对照，重点）**：Go **没有 `public`/`private`/`protected` 关键字**。可见性完全由**标识符首字母大小写**决定：大写 = 导出（public），小写 = 包内私有（private，等价于 Java 的包私有/私有）。这是 Go 唯一的作用域控制手段，连 struct 的字段也一样（`User.Name` 大写才导出）。习惯了 Java 的访问修饰符体系要重新建立直觉（Go 没有 `protected`，也没有精细的包级/子类级控制）。

### 5.2 什么时候用包级变量（学习阶段 vs 工程阶段）

学习阶段你会经常见到包级变量，但工程里要谨慎：

更推荐的包级变量用法：

- 只读的固定值（例如一些默认值）
- 错误变量：`var ErrX = errors.New("...")`
- 只读的查表数据（map/slice），且不在运行中修改

不推荐的包级变量用法：

- 可变的全局状态（并发不安全、测试困难、依赖隐式）

工程上的常见替代：

- 把状态放进 struct 里，通过构造函数创建实例（让依赖显式）
- 通过参数传递依赖（函数式/依赖注入）

### 5.3 跨包“全局变量”最常见的实战：哨兵错误（sentinel error）

Go 工程里最常见、最推荐的一类“跨包包级变量”就是：**导出的错误哨兵**，用于让调用方可判断错误类型，而不是靠字符串判断。

在本仓库 `internal/01-basics` 里，你已经见过这种写法（见 [basics.go](./basics.go)）：

```go
package basics

import "errors"

var ErrNegativeN = errors.New("n must be >= 0")
```

调用方跨包使用时，一般写：

```go
if errors.Is(err, basics.ErrNegativeN) {
	// 处理“参数为负数”的这类错误
}
```

关键点：

- **不要**用 `err.Error()` 字符串做逻辑判断（文本可能变化，且容易误判）
- **不要**在调用方写 `errors.New("n must be >= 0")` 来对比（即使文本相同也不是同一个错误值）
- 如果错误在向外传递时被包装（`fmt.Errorf("...: %w", err)`），用 `errors.Is` 仍然能正确匹配到哨兵错误

> ⚠️ **注意**：哨兵错误是**包级导出的错误变量**，必须用 `errors.Is(err, ErrX)` 比较，不能 `==`（被 `%w` 包装后 `==` 会失败）。`errors.Is` 会顺着包装链一路解包匹配。这套机制对应 Java 的自定义异常类型判断（`if (e instanceof MyException)` 或 catch 分支），但 Go 没有异常/枚举，靠"比较错误值"实现分类。

你可以直接看对应的测试用例：[basics_test.go](./basics_test.go)（`TestFactorialUint64` 等验证了 `errors.Is(err, ErrNegativeN)` / `ErrOverflow`）。

## 6. 常量（const）

常量（constant）是“编译期就确定”的值，不能修改：

```go
const Pi = 3.14159
const AppName = "GoJuniper"
```

### 6.1 const 与 var 的关键区别

- `var`：运行期有内存地址，可变
- `const`：编译期替换，不可变，通常没有“可取地址”的概念

因此，`const` 常用于：

- 枚举（状态码、选项）
- 固定配置（默认值、版本号）
- 位运算标志（flag）

### 6.2 常量表达式的限制

常量必须由“常量表达式”构成，不能调用运行期函数：

```go
const A = 1 + 2     // OK
// const B = time.Now() // 编译错误：不是常量
```

> ⚠️ **注意（与浮点陷阱联动）**：因为常量在编译期用**任意精度**运算，`const x = 0.1 + 0.2` 后 `x == 0.3` 是 **true**——这和运行时变量的浮点误差（见 [Go基本数据类型.md](./Go基本数据类型.md) 4.7 节）完全相反！记住：**常量精确、变量有浮点误差**。这是《Learning Go》里最容易让人困惑的"同一个表达式两种结果"现象。

### 6.2.1 数字字面量写法（下划线 / 0b / 0o / 0x）

常量也支持多种进制与可读性分隔符：

```go
const (
	 Million = 1_000_000  // 千分位下划线，纯可读性
	 Bin     = 0b1010     // 二进制
	 Oct     = 0o777      // 八进制（推荐 0o 前缀）
	 Hex     = 0xFF       // 十六进制
)
```

> ⚠️ **注意**：`0777`（老写法）在 Go 里是**八进制**，极易和十进制混淆，Go 1.13+ 推荐 `0o777`。下划线只能放数字之间，编译期直接忽略。详见 [Go基本数据类型.md](./Go基本数据类型.md) 4.5 节。

### 6.3 未类型常量（untyped constant）

这是 Go 的强项：常量可以不绑定具体类型，在使用时自动适配。

```go
const x = 10 // untyped

var a int = x
var b int64 = x
var c float64 = x
```

对比：变量会固定类型，不能随意赋给别的整数类型。

```go
var x = 10 // x 是 int
// var a int64 = x // 编译错误
```

> ⚠️ **注意（Go 最反直觉的特性之一）**：`const x = 10` 是**未类型常量**，没有固定类型，赋给 `int`/`int64`/`float64` 都可以。但 `var x = 10` 一旦声明，`x` 就被定型为 `int`，不能再赋给 `int64`。未类型常量更灵活，类型化常量更安全（避免隐式转换 bug）。这是"声明 vs 常量"的核心差异。

### 6.4 iota（枚举）

```go
const (
	StatusInit = iota // 0
	StatusRunning     // 1
	StatusDone        // 2
)
```

常见写法：跳过 0（让默认值更安全）

```go
const (
	_ = iota
	StatusRunning
	StatusDone
)
```

> ⚠️ **注意（重点）**：`iota` **不是全局计数器，而是 const 块内的"行号"**。每个 `const (...)` 块里，iota 从 0 开始，每换一行 +1，离开块就归零。两个独立的 const 块里各自的 iota 互不影响。`StatusInit=iota` 之所以是 0，就是因为它在该块第 0 行。详见 [Go基本数据类型.md](./Go基本数据类型.md) 与 `enums.go`。

### 6.5 iota + 位运算（flags）

```go
const (
	FlagRead  = 1 << iota // 1
	FlagWrite             // 2
	FlagExec              // 4
)
```

> ⚠️ **注意**：位标志判断的"黄金写法"是 `perm & FlagRead != 0`，**不是** `== FlagRead`（后者要求其他位都为 0，组合权限时会误判）。`1 << iota` 生成 1/2/4/8 这种互不重叠的位，才能用 `|` 组合、`&` 检测。对应 Java 的 `EnumSet` 或位掩码（如 `1 << flag.ordinal()`）。

## 7. 声明 vs 赋值（必须分清）

- 声明（declare）：第一次引入名字
  - `var x int`
  - `x := 1`
- 赋值（assign）：给已存在的变量改值
  - `x = 2`

## 8. Go 的函数（func）怎么写：几个必须会的例子

### 8.1 最基础：输入参数 + 返回值

```go
func Sum(a, b int) int {
	return a + b
}
```

### 8.2 多返回值：Go 的标配风格

Go 很常见的写法是“结果 + 是否成功/错误”：

```go
func Max(xs []int) (int, bool) {
	if len(xs) == 0 {
		return 0, false
	}
	m := xs[0]
	for _, v := range xs[1:] {
		if v > m {
			m = v
		}
	}
	return m, true
}
```

这也是你在标准库里经常看到的模式（例如 map 的读取 `v, ok := m[k]`）。

> ⚠️ **注意（Java 对照）**：Go 的 `(v, ok)` 双返回值是 `Optional<T>` 的等价物——`ok==false` 表示"无值"，`ok==true` 时第一个值是真值。但 Go 没有编译期强制你检查 `ok`，忘了检查也不会报错（只是拿到零值），这点和 Java 的 `Optional.ifPresent()` / `orElseThrow()` 强制处理缺席值不同，需要自律。

### 8.3 命名返回值（新手先谨慎使用）

命名返回值会把返回值当作函数内的变量：

```go
func Divide(a, b int) (q int, ok bool) {
	if b == 0 {
		return 0, false
	}
	return a / b, true
}
```

初学阶段建议你优先写“显式 return 值”，减少误读。

### 8.4 变参函数（...）

```go
func SumAll(xs ...int) int {
	total := 0
	for _, v := range xs {
		total += v
	}
	return total
}
```

调用时：

```go
_ = SumAll(1, 2, 3)
xs := []int{1, 2, 3}
_ = SumAll(xs...) // slice 展开
```

> ⚠️ **注意**：变参 `...int` 必须放在**参数列表最后一位**，且一个函数最多一个变参。函数体内它就是普通 `[]int`。`xs...` 是把切片"展开"成多个实参，反过来 `SumAll(1,2,3)` 时 Go 自动把实参打包成切片。详见 [函数进阶_closures_defer.md](./函数进阶_closures_defer.md)。

### 8.5 defer：退出前执行（常用于关闭资源）

```go
func demo() error {
	f, err := os.Open("a.txt")
	if err != nil {
		return err
	}
	defer f.Close()
	return nil
}
```

记住一句话：defer 通常用于“打开资源后立即写上关闭”，避免忘记释放。

> ⚠️ **注意**：`defer` 的注册时机是**调用那一刻**，但执行时机是**函数 return 前**。它遵守 LIFO（后注册先执行）。更深的坑（参数求值时机、能改命名返回值）见 [函数进阶_closures_defer.md](./函数进阶_closures_defer.md) 与 `funcsx.go` 的 `DeferInspect`。

## 9. Go 的包（package）与引用（import）：从新手到能跑起来

### 9.1 一个 Go 文件的最小结构

```go
package main

import "fmt"

func main() {
	fmt.Println("hello")
}
```

### 9.2 import 的两种写法

单行：

```go
import "fmt"
```

多行（更常见）：

```go
import (
	"fmt"
	"strings"
)
```

### 9.3 引用你自己仓库里的包：用模块路径（go.mod 的 module）

本仓库的 module 是 `gojuniper`，所以导入路径形如：

```go
import "gojuniper/internal/01-basics"
```

然后就可以调用导出的标识符（大写开头）：

```go
fmt.Println(basics.Sum(1, 2))
```

你可以直接看现成的可运行例子：[cmd/hello/main.go](../../cmd/hello/main.go)

### 9.4 import 别名（alias）

当包名冲突或你想让代码更清楚时可以用别名：

```go
import b "gojuniper/internal/01-basics"

func main() {
	_ = b.Sum(1, 2)
}
```

### 9.5 internal 目录的特殊规则（你现在就用上了）

Go 里 `internal/` 是“限制导入范围”的机制：

- 只有位于 `internal` 父目录范围内的代码才能导入它
- 这能保证这些学习包只在本仓库内部使用，避免被别的项目直接依赖

> ⚠️ **注意**：`internal/` 是 Go 工具链**强制**的目录约定——放在 `internal/` 下的包，外部模块根本无法 import（编译报错）。这不是风格建议，是编译器规则。本仓库所有学习包都藏在 `internal/` 下，正是为了不被外部依赖。

## 10. 初始化顺序：包级变量与 init()

你可以用这个“足够正确的简化规则”理解 Go 初始化：

1. 先初始化依赖包（import 的包）
2. 再初始化当前包的：
   - 包级变量（按依赖关系决定顺序）
   - `init()` 函数（同一个包可以有多个 init，依次执行）
3. 最后进入 `main.main()`

### 10.1 init() 什么时候用

学习阶段会遇到 `init()`，但工程中不要滥用，因为它会让初始化变得“隐式”。

更推荐显式构造：

```go
func NewService(...) *Service { ... }
```

> ⚠️ **注意**：`init()` 不能手动调用、没有参数和返回值，且同一个包可以有多个（按文件名字典序执行）。它适合做"包级一次性设置"（如注册驱动 `sql.Register`）。但**不要在 init 里做有副作用、依赖顺序、或可能失败的初始化**——失败了只能 panic，且难以测试。工程上优先用显式 `NewXxx()` 构造函数。
